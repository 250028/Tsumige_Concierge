## 1. 調査・相談で使ったプロンプト

### Step1：文脈の共有（最初の1回）

```
# 前提
作るアプリの企画書

（プロジェクト内の企画書1_250028.md）
```

> 以降はアプリ名や目的を毎回書かなくて済むようにした。

### Step2：レイアウト調査

```
# 調査
この企画書を見て、このジャンルのアプリでよくあるメイン画面の
レイアウトを2～3個、特徴つきで。

# 相談
メイン画面に置く要素とおすすめレイアウトを箇条書きで。
コードはまだ不要。
```

→ AIが3つのレイアウト案（A: ダッシュボード型 / B: 今日の1本+チャット型 / C: カードスワイプ型）をビジュアル付きで提示してくれた。

### Step3：3つのいいとこどり

```
この3つのいいとこどりでもう一度
```

→ 短い指示でも前の文脈が引き継がれ、A・B・C の要素を統合したスマホモックを出力してくれた。

### Step4：スマホ vs PC の比較相談

```
# 相談
スマホ想定の方がいい？
PC想定だとどんな感じ？
```

→ スマホ版とPC版（3カラムレイアウト）を比較してもらい、それぞれのメリット・デメリットを整理した。

### Step5：方針決定

```
スマホファーストで作り、レスポンシブでPC対応で行こうと思う
```

## 2. モック生成で使ったプロンプト

```
# 依頼
相談したレイアウトでメイン画面のモックを作成

# 条件
* HTML / CSS(Bootstrap) / JS
* 計算やデータ保存のJSは不要（見た目だけ）

# ファイル構成
mock/
├── index.html
├── css/style.css
└── js/app.js
```

→ Bootstrap 5 + Bootstrap Icons を使ったスマホファーストのモックが生成された。ボトムタブ・ヒーローカードスタック・AIチャットストリップ・ゲームリストプレビューがすべて含まれており、PCレスポンシブも込みで出力された。

その後、CSSをBootstrapに切り替えるよう追加指示した。

```
CSSをBootstrapに変更
```

→ カスタム CSS を約300行から79行に削減。Bootstrap で代替できない部分（カードスタック回転・チャットバブル・ボトムナビ・PCグリッドレイアウト）だけを別ファイルに分離した。

## 3. AIの出力に自分が加えた変更

- **ゲームタイトルのデータ**：AIが生成したダミーデータを、実際に自分が積んでいるゲーム（ゼルダTotK / モンハンワイルズ / ペルソナ5 / スプラ3 / あつ森 / Elden Ring）に差し替えた
- **カラーパレット**：最初の出力がデフォルトBootstrapブルーに寄っていたため、「紫系（#7C3AED）とゴールド（#D97706）を使ってゲームっぽい雰囲気に」と追加指示して変更した
- **チャットの初期メッセージ**：AIが汎用的な例文を入れていたので、執事ペルソナらしい口調（「ご帰還なさいませ」）に書き直した
- **積みゲー城タブ**：AIのデザイン案にはなかった「達成済み / 未達成」の実績グリッドを自分で追加した
- **app.js のランダム関数**：AIはスマホ用ボタン1つしか実装していなかったが、スマホ用・PC用の2ボタンに対応するよう `randomGame()` を関数化して共通利用できるよう修正した
- **CSSの分離**：AIが `<style>` タグにまとめて出力したカスタムCSSを `css/style.css` に分離するよう指示した

## 4. AIを使ってみて気づいたこと

**うまくいった指示**

- 最初に企画書を丸ごと貼って文脈を渡したことで、その後の指示が短くても意図が伝わった。「この3つのいいとこどりでもう一度」という短い指示だけで統合されたモックが出てきた
- 「コードはまだ不要」と明示したことで、余計なコードが出てこず相談に集中できた。設計を固めてからコードを依頼するフローが効率よかった
- 「スマホ想定の方がいい？PC想定だとどんな感じ？」のように2択で比較を頼むと、メリット・デメリットをセットで出してくれるので意思決定しやすかった

**うまくいかなかった指示・気づいた注意点**

- 最初の調査プロンプトで「企画書を見て」とだけ書いたら企画書が共有されていなかったため空振りした。文脈の共有は必ずセットにする必要があることを学んだ
- AIが生成したゲームタイトルに架空のものが混じっていた。実在のゲーム名を使いたい場合は自分でデータを確認・書き換える必要がある（ハルシネーション対策）
- カラーの指定は「ゲームっぽい」などの曖昧な表現より、「#7C3AED の紫を primary に」と具体的なHEX値で伝えた方が意図通りの出力が得やすかった

## 機能一覧

### Must（必須・合格ライン）
- 積みゲーマイリストのCRUD（タイトル・ジャンル・プラットフォーム・ステータス・進捗メモの登録・一覧・編集・削除）
- ゲームステータス管理（未開封 / 序盤で放置 / 中断中 / プレイ中 / クリア済み）
- ユーザー認証（ログイン・ログアウト・新規登録）
- AIコンシェルジュチャット（「今から2時間暇」「スカッとしたい」などの入力に対しリストから提案）
- AIによる「今日の1本」おすすめ表示（ホーム画面のヒーロー枠）

### Should（できれば実装）
- おまかせランダムセレクト（AIが演出つきで1本を決定）
- あらすじ・モチベーター機能（ゲーム名を入力するとAIがストーリー要約を表示）
- 操作方法一覧表示（ゲームとプラットフォームを選ぶとAIが主要ボタン配置を一覧で出力）
- AIペルソナ切り替え（執事 / ゲーマー仲間 / ゲームの妖精）
- ゲーミフィケーション：クリアごとにポイント付与・「積みゲー城」の見た目が回復していく演出
- ゲーム情報の自動補完（タイトル入力時にAIがジャンル・魅力を提案しタグ付け）

## 画面遷移（どの画面からどこへ動くか）

```
[ログイン / 新規登録]
        ↓ ログイン成功
    [ホーム] ─────────────────────────────────────────┐
       │                                               │
  タブ: リスト    タブ: AIチャット   タブ: 城   タブ: プロフィール
       │                                               │
[リスト一覧]                                       [プロフィール]
       │                                               │
   カードタップ                                    ログアウト
       │                                               ↓
[ゲーム詳細]                                    [ログイン画面]
       │ 編集ボタン
       ↓
[ゲーム編集]
       │ 保存
       ↓
[ゲーム詳細] ← 戻る → [リスト一覧]

[ホーム] → ＋ボタン → [ゲーム登録] → 保存 → [ホーム]
```

### 遷移のポイント
- すべての画面にボトムタブを常時表示し、どこからでもホーム・リスト・チャット・城・プロフィールへ戻れる（行き止まりなし）
- ゲーム詳細画面は「< 戻る」でリスト一覧に戻る
- ゲーム登録・編集画面は「キャンセル」で元の画面に戻る
- 3タップ以内で目的の操作に到達できるよう設計

### メニュー方式：ボトムタブナビゲーション（5タブ）

```
[ ホーム ]  [ リスト ]  [ チャット ]  [ 🏰城 ]  [ プロフィール ]
```

採用理由：スマホファーストのため親指で届きやすい画面下部にナビを置いた。PCレスポンシブ時は左サイドバーナビに切り替え（Bootstrap の `d-none d-md-flex` で制御）。
