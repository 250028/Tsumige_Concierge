# 設計書

## アプリ名
積みゲー・コンシェルジュ

---

## 機能一覧（企画書から整理）

### Must（必須・合格ライン）
- 積みゲーマイリストのCRUD（タイトル・ジャンル・プラットフォーム・ステータス・進捗メモの登録・一覧・編集・削除）
- ゲームステータス管理（未開封 / 序盤で放置 / 中断中 / プレイ中 / クリア済み）
- ユーザー認証（ログイン・ログアウト・新規登録）
- AIコンシェルジュチャット（「今から2時間暇」「スカッとしたい」などの入力に対しリストから提案）
- AIによる「今日の1本」おすすめ表示（ホーム画面のヒーロー枠）

### Should（できれば実装）
- おまかせランダムセレクト（AIが演出つきで1本を決定）
- あらすじ・モチベーター機能（ゲーム名を入力するとAIがストーリー要約を表示）
- 操作方法一覧表示（ゲームとプラットフォームを選ぶとAIが主要ボタン配置を一覧で出力）
- AIペルソナ切り替え（執事 / ゲーマー仲間 / ゲームの妖精）
- ゲーミフィケーション：クリアごとにポイント付与・「積みゲー城」の見た目が回復していく演出
- ゲーム情報の自動補完（タイトル入力時にAIがジャンル・魅力を提案しタグ付け）

## 画面遷移

```
[ログイン / 新規登録]
        ↓ ログイン成功
    [ホーム] ─────────────────────────────────────────┐
       │                                               │
  タブ: リスト    タブ: AIチャット   タブ: 城   タブ: プロフィール
       │                                               │
[リスト一覧]                                       [プロフィール]
       │                                               │
   カードタップ                                    ログアウト
       │                                               ↓
[ゲーム詳細]                                    [ログイン画面]
       │ 編集ボタン
       ↓
[ゲーム編集]
       │ 保存
       ↓
[ゲーム詳細] ← 戻る → [リスト一覧]

[ホーム] → ＋ボタン → [ゲーム登録] → 保存 → [ホーム]
```

### 遷移のポイント
- すべての画面にボトムタブを常時表示し、どこからでもホーム・リスト・チャット・城・プロフィールへ戻れる（行き止まりなし）
- ゲーム詳細画面は「< 戻る」でリスト一覧に戻る
- ゲーム登録・編集画面は「キャンセル」で元の画面に戻る
- 3タップ以内で目的の操作に到達できるよう設計

## メニュー方式

**ボトムタブナビゲーション（5タブ）**を採用。

```
[ ホーム ]  [ リスト ]  [ チャット ]  [ 🏰城 ]  [ プロフィール ]
```

### 選んだ理由
- スマホファーストのため、親指で届きやすい画面下部にナビを置く
- 5つの主要機能が常時見えることで「次に何ができるか」が直感的にわかる
- PCレスポンシブ時はサイドバーナビに切り替え（`md:` ブレークポイントで切り替え）

## メイン画面のモック（HTML/CSS/JS）

- **コードの保存先：** `mock/index.html` / `mock/css/style.css` / `mock/js/app.js`
- **実装した画面：** ホーム画面（スマホ版）・PCレイアウト版

### 画面の動き（JS で実装した切り替え・インタラクション）
- ボトムタブのタブ切り替え（ホーム / リスト / チャット / 城 / プロフィール）
- 「今すぐプレイ」「ランダム」「スキップ」ボタンのクリックイベント
- チャット入力欄への送信動作（ダミーレスポンス表示）

### 工夫した点
1. **カードスタック演出（C レイアウトから採用）**
   AIのおすすめカードを重なった複数枚のカードで表示し、「次の候補がある」ことを視覚的に伝える。スワイプ感覚をボタン3つ（プレイ・ランダム・スキップ）で再現。

2. **AI チャットを常にすぐ使える位置に（B レイアウトから採用）**
   ホーム画面にチャット入力欄を埋め込み、別ページに移動しなくてもコンシェルジュに話しかけられる。最初から会話がある状態にして「使い方がわかる」状態にした。

3. **ダッシュボード型の統計バッジ（A レイアウトから採用）**
   ヘッダーに「積み 24 / 済 12」のバッジを小さく表示し、全体把握を1秒でできるようにした。

4. **レスポンシブ：ボトムタブ → サイドバー切り替え**
   スマホ（〜767px）：ボトムタブナビ / PC（768px〜）：左サイドバーナビ + 3カラムレイアウト

## 保存するデータ（テーブルの形）

### `users` テーブル
| カラム名 | 型 | 説明 |
|---|---|---|
| id | INT (PK) | ユーザーID |
| name | VARCHAR(100) | ユーザー名 |
| email | VARCHAR(255) | メールアドレス |
| password | VARCHAR(255) | ハッシュ化パスワード |
| persona_type | ENUM | AIペルソナ設定（butler / gamer / fairy） |
| points | INT | 累計ポイント |
| created_at | DATETIME | 登録日時 |

### `games` テーブル（中心テーブル）
| カラム名 | 型 | 説明 |
|---|---|---|
| id | INT (PK) | ゲームID |
| user_id | INT (FK) | 所有ユーザー |
| title | VARCHAR(255) | ゲームタイトル |
| genre | VARCHAR(100) | ジャンル（RPG / ACT / STG など） |
| platform | VARCHAR(50) | プラットフォーム（Switch / PS5 / PC など） |
| status | ENUM | 未開封 / 序盤で放置 / 中断中 / プレイ中 / クリア済み |
| progress_note | TEXT | 進捗メモ（「第3の祠まで終了」など） |
| purchase_date | DATE | 購入日 |
| created_at | DATETIME | 登録日時 |
| updated_at | DATETIME | 更新日時 |

### `chat_logs` テーブル
| カラム名 | 型 | 説明 |
|---|---|---|
| id | INT (PK) | ログID |
| user_id | INT (FK) | ユーザー |
| role | ENUM | user / assistant |
| message | TEXT | メッセージ内容 |
| created_at | DATETIME | 送信日時 |

### テーブル間のリレーション
```
users (1) ──< games (多)     ← ユーザーは複数の積みゲーを持つ
users (1) ──< chat_logs (多) ← ユーザーは複数のチャット履歴を持つ
```

## レスポンシブ対応方針

| 画面幅 | レイアウト | ナビ |
|---|---|---|
| ～767px（スマホ） | 1カラム、縦スクロール | ボトムタブ（5項目） |
| 768px〜（PC・タブレット） | サイドバー(200px) + メイン + チャットパネル(280px) | 左サイドバー |

- スマホ優先で実装し、`@media (min-width: 768px)` でPC用レイアウトを上書き
- Next.js + CSS Modules（または Tailwind）で管理
- ボトムタブは PC では `display: none`、サイドバーはスマホでは `display: none`


## 使う技術スタックと、その理由

| 層 | 技術 |
|---|---|
| フロントエンド | Next.js (App Router) + TypeScript |
| CSS | Tailwind CSS v4 |
| バックエンド | Next.js API Routes（同一プロジェクト） |
| ORM | Prisma |
| データベース | MySQL |
| AI | Gemini API |
| 開発環境 | Node.js + npm/pnpm・VSCode・GitHub |

### 採用理由
- Next.js：フロントとAPIを1プロジェクトで管理でき、初学者にシンプルな構成になるため
- TypeScript：型安全により実装ミスを早期発見できるため
- Tailwind CSS：`md:` ブレークポイントでスマホ・PCのレスポンシブ対応が書きやすいため
- Prisma：schema.prismaでテーブル設計を一元管理でき、SQLを直接書かずTypeScriptで
  DB操作できるため。`npx prisma migrate dev` でMySQLのテーブル自動生成も可能
- MySQL：授業で学んだ技術を活かせる・リレーショナル設計に適しているため
- Gemini API：テキスト生成・チャット両対応で、無料枠が充実しているため
