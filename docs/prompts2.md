## 2. ORMの仕組みを理解するために使ったプロンプト

```
ORMの仕組みが分からない
```

### AIの出力の要点
- ORMとは「TypeScriptのメソッドでDBを操作できる橋渡し役」と説明された
- SQLを直接書く場合とORMを使う場合のコードを並べて比較してもらった

- Prismaでは `schema.prisma` にテーブル構造を書いて `npx prisma migrate dev` を実行するだけでMySQLにテーブルが自動生成されると理解した

## 3. モック画面の移植についての相談

```
モック画面をhtmlとCSSはBootstrapで作成したけど、
本格的にアプリ作成するときにはNext.jsとTailwind CSSで作りたい。
ほぼモック画面と同じようになる？
```

### AIの出力の要点
- 見た目の再現度はほぼ100%可能と回答された
- BootstrapとTailwindは役割が同じ（CSSフレームワーク）なので外見の差はほとんど出ない
- 変わるのは書き方と構造：
  - Bootstrapのクラス → Tailwindの細かいクラスの組み合わせに変わる
    - 例：`btn btn-primary` → `bg-blue-500 text-white px-4 py-2 rounded`
  - 1つのHTMLファイルで `display: none` でタブ切り替えしていたものが、Next.jsではURLベースのページ遷移になる
  - `index.html` → `app/page.tsx`（ホーム）、`app/list/page.tsx`（リスト）のようにファイルが分かれる
- 見た目ではなく「データが本物になる」のが最大の変化

### 自分が加えた変更・判断
- モックの設計はそのまま活かせると確認できたので、Next.jsでの実装時もモックのレイアウトを参考にする

# API設計メモ：ゲーム情報の取得元について

## 質問

> アプリでできることに「放置していたゲームの『前回までのあらすじ』やAIに聞くことで、スムーズにプレイを再開できる」「プレイを再開したいゲームの『プラットフォーム（Switch/PS5/PCなど）』を指定するだけで、AIがそのゲームの基本操作方法やボタン配置を分かりやすく一覧で教えてくれる」とあるが、どこからゲームの情報を引っ張ってくる？

---

## 選択肢の整理

### 選択肢A：AIの学習済み知識に丸ごと頼る（最もシンプル）

Gemini APIに「ゼルダの伝説 ブレス オブ ザ ワイルドのSwitchでの基本操作を教えて」とそのまま聞く方法。

| | 内容 |
|---|---|
| **メリット** | 外部APIの契約不要、実装が簡単 |
| **デメリット** | ハルシネーション（嘘の情報）のリスクがある、マイナーなゲームは精度が落ちる |

### 選択肢B：外部のゲームDBと組み合わせる

**RAWG API**（無料）や **IGDB API**（Twitch認証必要）などのゲームデータベースからタイトル・ジャンル・画像などを取得して、それをAIに渡す方法。

```
ユーザー入力「ゼルダBotW」
    → RAWG APIでゲーム情報取得
    → その情報 + 「操作方法を教えて」をGeminiに渡す
    → 精度の高い回答を返す
```

| | 内容 |
|---|---|
| **メリット** | ハルシネーション減少、ゲームのサムネ画像も取れる |
| **デメリット** | API連携の実装が増える、RAWGは操作方法データは持っていない |

### 選択肢C：ユーザー自身が情報を入力＋AIが補助

ゲーム登録時にユーザーがタイトルを入力したら、AIが「このゲームのジャンルはRPGですか？」と確認しながら情報を補完していく方式。

---

## 結論・方針

### 基本方針：選択肢A（Gemini API）ベース＋RAWG APIを補助で追加

> 「操作方法やあらすじはAIが得意な領域なので、まずは選択肢Aで始めて、プロンプトで精度を上げるのが現実的」

プロンプトの工夫例：

```
「{ゲームタイトル}の{プラットフォーム}版の操作方法を教えてください。
 もし確実な情報がない場合は、「情報が不確かです」と明示してください。」
```

---

## RAWGとGemini APIの役割分担

### フロー①：ゲーム登録時

```
ユーザー（タイトル入力）
  → アプリ（タイトルで検索）
  → RAWG API（ゲーム基本情報を返す）
  → アプリにジャンル・画像・クリア時間が自動入力される
```

### フロー②：AIコンシェルジュへの質問時

```
ユーザー（「2時間暇、スカッとしたい」）
  → アプリ（積みゲーリスト＋条件を渡す）
  → Gemini API（条件＋リストをもとに提案）
  → おすすめ1本 ＋ 理由 ＋ あらすじ・操作方法を返す
```

### 各APIが担当する範囲

| API | 担当 |
|---|---|
| **RAWG API** | ゲーム基本情報、カバー画像、ジャンル・評価、推定プレイ時間（※ない場合はGeminiが補完） |
| **Gemini API** | あらすじ要約、操作方法の説明、気分マッチング、ペルソナチャット |

### 登録フォームのイメージ

```
タイトル入力 → 「検索」ボタン → RAWGから候補を取得
→ ユーザーが選ぶ → ジャンル・画像・評価が自動入力される
```

1. まず **Gemini APIチャット部分** を作る（コアな価値はここ）
2. 余裕が出たら **RAWG APIを後から追加** する（登録UXの改善）

# RAWG APIの日本語対応について

## 質問

> RAWG APIって海外だけど、日本語に直したり出力できる？

---

## 結論

**できる**。ただし方法によって精度が変わる。

---

## RAWGのデータ自体について

RAWGは基本的に英語データがメイン。日本語対応は部分的で、ゲームによってバラつきがある。

| データ項目 | 日本語の有無 |
|---|---|
| ゲームタイトル | 日本語表記があるゲームもあるが不安定 |
| ジャンル | 英語のみ（RPG, Action など） |
| 説明文 | ほぼ英語のみ |
| 画像 | 言語関係なし |

---

## 日本語化の方法（3パターン）

### パターン①：Geminiに翻訳させる（一番かんたん）

RAWGから英語で取得した後、そのままGeminiに渡して翻訳する。

```
RAWGの結果 → アプリ → Geminiに「以下を日本語に翻訳して」→ 表示
```

登録時に1回だけやればいいのでAPI呼び出しも最小限で済む。

### パターン②：ジャンルだけ自前でマッピングする（コスト節約）

ジャンルはRAWGで使われる単語が限られているので、自分で対訳テーブルを作る方法。

```javascript
const genreMap = {
  "RPG": "RPG",
  "Action": "アクション",
  "Shooter": "シューター",
  "Adventure": "アドベンチャー",
  "Strategy": "ストラテジー",
  // ...
}
```

翻訳APIを呼ばずに済むので速いし無料。

### パターン③：日本語タイトルはユーザー入力に頼る

RAWGで英語タイトルを取得しつつ、「日本語タイトル（任意）」という入力欄を別途用意する。ユーザーが自分で「ゼルダの伝説」と入力できるようにしておく形。

---

## このアプリへの推奨

RAWGから取れる日本語化が必要なのは実質**ジャンルと説明文くらい**で、画像やスコアはそのまま使える。

| 項目 | 対応方法 |
|---|---|
| ジャンル | パターン②（対訳テーブル） |
| 説明文・あらすじ | どうせGeminiが要約・生成するのでRAWGの説明文は使わなくてよい |

この切り分けが一番シンプル。
