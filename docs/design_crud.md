## 0. スコープ

| 項目 | 今回やる | 今回やらない（後回し） |
|---|---|---|
| ゲーム登録 | 手動入力での登録 | RAWG API連携（タイトル検索→ジャンル・カバー画像自動補完） |
| ゲーム一覧 | 一覧表示・ステータスでフィルタ | 並び替え・検索 |
| ゲーム詳細・編集 | 閲覧・全項目編集 | プレイセッション（タイマー）連携 |
| ゲーム削除 | 削除（確認モーダルあり） | 論理削除（今回は物理削除でOK） |
| 画像 | カバー画像なし（グレーのダミー表示） | 画像アップロード |

RAWG連携・画像アップロードは、CRUDの土台ができてから別ブランチで追加する。

## 2. APIルート設計

ベースパス: `/api/games`
認証: すべてのエンドポイントで `lib/session.ts` のセッションを確認し、`userId` を取得する。未ログインは `401`。
所有者チェック: 対象の `Game.userId` がセッションの `userId` と一致しない場合は `403`（他人のゲームは操作不可）。

### 2.1 一覧取得 `GET /api/games`

| 項目 | 内容 |
|---|---|
| クエリパラメータ | `status`（任意・`GameStatus` のいずれか。指定時はそのステータスのみ絞り込み） |
| 処理 | セッションの `userId` に紐づく `Game` を `createdAt` 降順で取得 |
| レスポンス 200 | `Game[]`（配列） |
| レスポンス 401 | 未ログイン |

### 2.2 登録 `POST /api/games`

| 項目 | 内容 |
|---|---|
| リクエストボディ | `title`(必須, string), `genre`(任意), `platform`(任意), `status`(任意, デフォルト `未開封`), `purchaseDate`(任意, `YYYY-MM-DD`), `progressNote`(任意) |
| バリデーション | `title` が空文字/未指定なら `400`。`status` が `GameStatus` enum に無い値なら `400` |
| 処理 | セッションの `userId` を付与して `Game` を作成 |
| レスポンス 201 | 作成した `Game` |
| レスポンス 400 | バリデーションエラー（`{ error: string }`） |
| レスポンス 401 | 未ログイン |

### 2.3 詳細取得 `GET /api/games/[id]`

| 項目 | 内容 |
|---|---|
| 処理 | `id` で `Game` を取得。存在しなければ `404`。`userId` 不一致なら `403` |
| レスポンス 200 | `Game` |

### 2.4 更新 `PUT /api/games/[id]`

| 項目 | 内容 |
|---|---|
| リクエストボディ | 登録時と同じ項目（部分更新ではなく全項目を送る想定。編集フォームは詳細取得した値をそのまま初期値にするため） |
| バリデーション | 登録時と同じ |
| 処理 | 所有者チェック後 `update`。`status` が `クリア済み` に変更された場合のみ `clearedAt` に現在時刻をセット。それ以外への変更では `clearedAt` を `null` に戻す |
| レスポンス 200 | 更新後の `Game` |
| レスポンス 400/401/403/404 | 上記と同様 |

### 2.5 削除 `DELETE /api/games/[id]`

| 項目 | 内容 |
|---|---|
| 処理 | 所有者チェック後、物理削除（`onDelete: Cascade` により `PlaySession` も連動削除される） |
| レスポンス 204 | 削除成功（本文なし） |
| レスポンス 401/403/404 | 上記と同様 |

## 3. 画面ごとの詳細設計

### 3.1 積みゲーリスト一覧（`/list`）

**表示項目（カードごと）**
- カバー画像（今回はダミー画像 or 頭文字アイコン表示）
- タイトル
- プラットフォーム
- ステータスバッジ（色分け：未開封=グレー / 序盤で放置=黄 / 中断中=オレンジ / プレイ中=紫(primary) / クリア済み=ゴールド(accent)）
- 進捗メモ（1行に省略表示）

**フィルターチップ**
- 「すべて」「未開封」「序盤で放置」「中断中」「プレイ中」「クリア済み」
- チップをタップ → `?status=xxx` をクエリに付けて再取得（Server Componentなので `searchParams` で受け取る）

**操作**
- カードタップ → `/games/[id]` に遷移
- 右下（or ヘッダー）に「＋ 登録」ボタン → `/games/new`

**空状態**
- 1件も無い場合は「積みゲーはまだ登録されていません」+登録導線を表示

### 3.2 ゲーム登録（`/games/new`）

**フォーム項目**

| 項目 | input type | 必須 | 備考 |
|---|---|---|---|
| タイトル | text | ○ | |
| ジャンル | text | - | 手動入力（RAWG連携までの暫定） |
| プラットフォーム | select | - | Switch / PS5 / PS4 / Xbox / PC / その他 |
| ステータス | select | - | デフォルト「未開封」 |
| 購入日 | date | - | |
| 進捗メモ | textarea | - | |

**動作**
1. 送信ボタン押下で `fetch('/api/games', { method: 'POST', body: JSON.stringify(form) })`
2. 成功（201）→ `/list` へ遷移
3. 失敗（400）→ フォーム上部にエラーメッセージ表示、画面遷移しない
4. 送信中はボタンを disabled にして二重送信を防ぐ

### 3.3 ゲーム詳細・編集（`/games/[id]`）

**表示モード（初期表示）**
- 登録項目をすべて表示（読み取り専用）
- 「編集」ボタン・「削除」ボタンを表示

**編集モード**
- 「編集」ボタン押下でフォームに切り替え（登録画面とほぼ同じ入力項目、既存値を初期値としてセット）
- 「保存」→ `PUT /api/games/[id]` → 成功したら表示モードに戻る（または `/list` に戻る）
- 「キャンセル」→ 表示モードに戻す（未保存の変更は破棄）

**削除**
- 「削除」ボタン押下で確認モーダル（「本当に削除しますか？この操作は取り消せません」）
- 確認後 `DELETE /api/games/[id]` → 成功したら `/list` へ遷移

## 4. エラーハンドリング方針

- API側はすべて `try/catch` で包み、想定外エラーは `500 { error: "サーバーエラーが発生しました" }` を返す（詳細なスタックトレースはクライアントに返さない）
- クライアント側は `fetch` の `!response.ok` を見てエラーメッセージを画面に表示する（`alert` は使わず、フォーム上部にエラー用のUIを置く）
- 認証切れ（401）を受け取った場合は `/login` にリダイレクトする
